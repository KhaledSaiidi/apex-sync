  Evidence pointing to a race / eventual-consistency issue:

  - garage-backup-bootstrap failed several times, then later succeeded without a manifest change.
  - The failed logs showed it reached Garage and started running Garage CLI commands.
  - After the failures, Garage already had:
      - key percona-backup-key
      - bucket percona-backups
      - later the Kubernetes secret percona-backup-garage-s3

  - Retiggering jobs one by one worked, which fits “first run partially bootstraps Garage state, later run observes that state and completes.”

  Most likely failure point is one of these immediately-after-create steps:

  /garage key create "$3"
  /garage bucket create "$2"
  /garage bucket allow --key "$9" --read --write --owner "$2"
  /garage key info --show-secret "$9"

  On a fresh Garage cluster, the key/bucket may be created but not immediately visible/usable to the next command. Since the script has set -e, any transient non-zero exit
  kills the hook. The later retry then succeeds because the key/bucket already exists.

  A fix can still be done envsubst-safe by keeping your positional style and adding retry wrappers that use $1, $2, etc., or escaping any shell vars as $$KEY_NAME if your
  envsubst setup supports that. I would not switch the script to plain $KEY_NAME variables in this repo.
